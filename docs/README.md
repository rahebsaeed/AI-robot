# Yahboom Rosmaster X3 PLUS AI Navigation Documentation

This document explains how the AI companion system is structured, how the launch script starts ROS and the AI stack, how perception and robot control communicate, how map search works, how RViz is opened, and how shutdown is handled.

Algorithm SVG schema: [algorithm_schema.svg](./algorithm_schema.svg)

## 1. System Goal

The project combines:

- Yahboom ROS navigation running inside `yahboom_container`.
- AMCL localization on a saved map, usually `salle_robotique`.
- Lidar obstacle data from `/scan`.
- RGB camera frames from `/camera/rgb/image_raw`.
- A Python AI companion on the host in `/home/jetson/AI/main.py`.
- YOLO object detection for visual search.
- A Qt robot face interface with buttons for stop search, RViz, and microphone mute.
- UDP bridges between ROS-in-Docker and the host AI process.

The core user workflow is:

1. Start `start_navigation.sh`.
2. The script starts ROS, localization, navigation, bridges, RViz, and the AI.
3. The user gives voice commands.
4. The AI either answers conversationally, moves the robot, navigates to saved places, or searches the map for objects.
5. Search uses map waypoints and live camera detection.
6. CTRL+C stops AI, robot motion, lidar drivers, ROS nodes, and saves the map.

## 2. Main Files

| File | Purpose |
| --- | --- |
| `start_navigation.sh` | Single launcher for Docker ROS, hardware bringup, navigation, AMCL initialization, bridges, RViz, AI startup, and cleanup. |
| `main.py` | Main AI loop. Handles voice commands, search routing, saved places, safe movement, face interface updates, and UI button commands. |
| `core/perceptions.py` | Microphone, Whisper speech recognition, YOLO object detection, UDP camera receiver, UDP lidar receiver, speech output. |
| `core/robot.py` | Robot movement over UDP, move_base goal publishing, AMCL pose reading, waypoint generation, RViz opening. |
| `core/search_tasks.py` | Map-based visual search algorithm. Resolves object names, chooses waypoints, checks live vision, handles stuck waypoints, and reports results. |
| `core/face_bridge.py` | Sends UDP messages to the Qt face interface. |
| `core/places_memory.py` | Stores named map positions for later navigation commands. |
| `robot_face.py` | Qt face UI, status display, and buttons. |

## 3. Runtime Architecture

The architecture has two main execution zones:

### Host / Jetson AI Zone

The host runs:

- `main.py`
- `robot_face.py`
- Whisper speech recognition
- YOLO inference
- UDP listeners for camera, lidar, and AMCL pose
- UDP sender for movement commands

### Docker / ROS Zone

The Docker container runs:

- `roscore`
- Yahboom hardware bringup
- Lidar driver
- Astra camera ROS stream
- `map_server`
- `amcl`
- `move_base`
- UDP bridge scripts generated by `start_navigation.sh`
- RViz when launched inside the container

The two zones communicate through UDP and ROS topics.

## 4. Ports and Data Flow

| Port | Direction | Meaning |
| --- | --- | --- |
| `5010` | Docker to host | Lidar summary from `/scan` to `Perceptions`. |
| `5020` | Host to Docker | AI movement command bridge to `/cmd_vel`. |
| `5030` | Docker to host | JPEG RGB camera frames from `/camera/rgb/image_raw`. |
| `5040` | Docker to host | AMCL pose from `/amcl_pose`. |
| `5006` | Face UI to main AI | UI button commands such as `stop_search`, `show_rviz`, `mic_off`, `mic_on`. |

## 5. Launch Sequence

`start_navigation.sh` performs this sequence:

1. Defines map, Docker container, robot IP, map path, and AI path.
2. Stops old AI and bridge processes.
3. Prepares X11 permissions for RViz and the face UI.
4. Restarts robot hardware service.
5. Restarts `yahboom_container`.
6. Syncs device links inside Docker:
   - `/dev/camera_head`
   - `/dev/camera_depth`
   - `/dev/camera_usb`
   - `/dev/ydlidar`
   - `/dev/myserial`
   - `/dev/myspeech`
7. Stops old ROS processes.
8. Checks that the map YAML exists.
9. Starts `roscore`.
10. Starts Yahboom body, lidar, and Astra camera launch.
11. Starts navigation with the selected map.
12. Waits for AMCL and move_base services.
13. Runs AMCL global localization if enabled.
14. Spins the robot for AMCL calibration.
15. Stops the robot after calibration.
16. Prints the current AMCL pose.
17. Clears move_base costmaps.
18. Waits for `/scan` and `/camera/rgb/image_raw`.
19. Starts UDP bridges:
    - lidar bridge
    - command bridge
    - RGB camera bridge
    - pose bridge
20. Starts RViz.
21. Starts the AI companion.

## 6. ROS Bridge Details

### Lidar Bridge

The lidar bridge subscribes to `/scan`, extracts valid ranges, and sends:

```json
{
  "front": 0.82,
  "all": 0.61
}
```

The host AI receives this on UDP port `5010`.

### Command Bridge

The command bridge receives host UDP commands on port `5020`:

```json
{
  "type": "move",
  "direction": "forward",
  "speed": 0.25
}
```

It converts that to `/cmd_vel` messages with capped linear and angular velocity.

### RGB Camera Bridge

The RGB bridge subscribes to `/camera/rgb/image_raw`, resizes frames to `320x240`, JPEG-encodes them, and sends packets to UDP port `5030`.

### Pose Bridge

The pose bridge subscribes to `/amcl_pose`, converts quaternion orientation to yaw, and sends:

```json
{
  "x": -1.25,
  "y": 4.80,
  "yaw": 1.57
}
```

The host AI caches this pose for fast distance checks.

## 7. Perception System

`core/perceptions.py` handles:

- Speech-to-text with Whisper.
- Microphone selection and calibration.
- UDP camera frame reception.
- UDP lidar distance reception.
- YOLO object detection.
- Text-to-speech with `espeak-ng`.

### Vision Data Format

The `see()` method returns a frame and a structured vision dictionary:

```python
{
    "camera_ok": True,
    "description": "I detect: bottle at center (medium, 0.82).",
    "objects": [
        {
            "name": "bottle",
            "confidence": 0.82,
            "bbox": [120.0, 44.0, 180.0, 210.0],
            "position": "center",
            "distance_hint": "medium",
            "source": "yolo11"
        }
    ],
    "source": "udp_camera",
    "frame_size": [320, 240],
    "frame_age": 0.12
}
```

## 8. Main AI Loop

`main.py` creates:

- `Brain`
- `Perceptions`
- `Robot`
- `FaceBridge`
- `PlacesMemory`
- `MapSearchTask`
- `UiCommandListener`

The loop does:

1. Read latest camera and lidar state.
2. Listen for a voice command.
3. Route command by priority:
   - save / rename / delete / list saved places
   - map search request
   - normal conversation or movement command
4. For search commands, bypass the language model and call `MapSearchTask.search()`.
5. For movement commands, apply safe movement rules before sending robot commands.
6. Update face UI and speech output.

## 9. Search Command Parsing

Search commands are detected by patterns such as:

- `search for`
- `look for`
- `find`
- `where is`
- `locate`
- `go find`

Examples:

| Voice command | Parsed target | Detector target |
| --- | --- | --- |
| `search for the bottle` | `bottle` | `bottle` |
| `find the bottle of water` | `bottle of water` | `bottle` |
| `where is the chair` | `chair` | `chair` |
| `find the phone` | `phone` | `cell phone` |

YOLO aliases are handled in `core/search_tasks.py`.

## 10. Map Search Algorithm

The map search algorithm uses:

- AMCL pose
- generated free-space waypoints
- move_base goals
- live YOLO camera checks
- stuck waypoint skipping
- optional local scans and waypoint checks controlled by environment variables

The current algorithm is designed to avoid wasting time at each waypoint. It checks the camera while the robot is moving. It only stops when:

- the target is detected and confirmed,
- the user cancels search,
- the whole search completes,
- shutdown is triggered,
- optional waypoint checking/scanning is enabled.

### Search Steps

1. Reset search state.
2. Validate the target text.
3. Resolve target to a YOLO class or alias.
4. Cancel any previous navigation goal.
5. Stop robot before starting new search.
6. Check current camera before waypoint generation.
7. If target is already visible, stop and report found.
8. Clear costmaps.
9. Generate search waypoints from the map.
10. Choose nearest unvisited waypoint from current AMCL pose.
11. Send a move_base goal.
12. While moving:
    - read camera detections,
    - if target appears, cancel navigation, stop, confirm detection, and report found,
    - read AMCL distance to goal,
    - if close enough, mark waypoint visited and continue,
    - if progress is stuck, mark waypoint skipped and continue.
13. Continue until target found or waypoints exhausted.
14. Report not found if all reachable points were searched.

### Stuck Waypoints

A waypoint is considered stuck when distance to the goal does not improve enough for `AI_SEARCH_STUCK_TIMEOUT`.

Default behavior:

- Do not rotate.
- Do not run manual recovery.
- Mark waypoint as skipped.
- Continue to the next waypoint.

Manual rotation can be re-enabled with:

```bash
export AI_SEARCH_ROTATE_ON_STUCK=1
export AI_SEARCH_MAX_RECOVERIES_PER_WP=1
```

## 11. Search Configuration

| Variable | Default | Meaning |
| --- | --- | --- |
| `AI_SEARCH_NAV_TIMEOUT` | `14.0` | Maximum seconds to spend on one waypoint. |
| `AI_SEARCH_STUCK_TIMEOUT` | `3.0` | Seconds without distance progress before skipping waypoint. |
| `AI_SEARCH_CONFIRM_READS` | `1` | Required target detections for confirmation. |
| `AI_SEARCH_CONFIRM_INTERVAL` | `0.08` | Delay between confirmation reads. |
| `AI_SEARCH_MAX_RECOVERIES_PER_WP` | `0` | Number of manual recoveries per waypoint. |
| `AI_SEARCH_WAYPOINT_CHECK_SECONDS` | `0.0` | Optional stopped camera check at a waypoint. Disabled by default. |
| `AI_SEARCH_SCAN_SECONDS` | `0.0` | Optional turn scan duration. Disabled by default. |
| `AI_SEARCH_START_SCAN_SECONDS` | `0.0` | Optional scan before waypoint search. Disabled by default. |
| `AI_SEARCH_FULL_SCAN_EVERY` | `0` | Optional full scan every N visited waypoints. Disabled by default. |
| `AI_SEARCH_STOP_AT_WAYPOINT` | `0` | Stop and check waypoint after navigation. Disabled by default. |
| `AI_SEARCH_APPROACH_ON_FOUND` | `0` | Move toward target after finding it. Disabled by default. |
| `AI_SEARCH_ROTATE_ON_STUCK` | `0` | Rotate left when stuck. Disabled by default. |
| `AI_SEARCH_SPACING_M` | `1.00` | Distance between generated search waypoints. |
| `AI_SEARCH_MAX_POINTS` | `90` | Maximum generated search waypoints. |
| `AI_SEARCH_CLEARANCE_M` | `0.50` | Minimum clearance from walls/obstacles in map waypoint generation. |

## 12. Waypoint Generation

`Robot.get_search_waypoints()` reads the map YAML and PGM image from:

```text
/root/yahboomcar_ws/src/yahboomcar_nav/maps
```

It:

1. Loads map resolution and origin.
2. Loads the occupancy image.
3. Dilates occupied pixels by a clearance radius.
4. Keeps only clearly free cells.
5. Converts free pixels to map-frame coordinates.
6. Downsamples to `AI_SEARCH_MAX_POINTS`.

This keeps goals away from walls and obstacles.

## 13. Robot Navigation

`Robot.goto_map(x, y, yaw)` publishes a goal to:

```text
/move_base_simple/goal
```

The goal frame is:

```text
map
```

The robot pose is read from cached UDP AMCL pose data when available. If the UDP cache is empty, `Robot.get_amcl_pose()` falls back to a one-shot Docker ROS read.

## 14. RViz Behavior

RViz is opened automatically by `start_navigation.sh` unless disabled:

```bash
export AI_AUTO_RVIZ=0
```

The face UI has an `RViz` button. When clicked:

1. `robot_face.py` sends `show_rviz` to UDP port `5006`.
2. `main.py` receives it in `UiCommandListener`.
3. `Robot.launch_rviz()` checks if RViz is already running.
4. If RViz exists, it tries to raise the existing window.
5. If no RViz exists, it starts one.

RViz displays:

- map
- global costmap
- robot model
- AMCL pose covariance
- lidar scan
- global plan
- TF tree

## 15. Shutdown Behavior

CTRL+C in `start_navigation.sh` runs `cleanup()`:

1. Stop host AI and face interface.
2. Stop robot motion and stale `/cmd_vel` publishers.
3. Stop lidar motor and lidar ROS drivers.
4. Save the map using `map_saver`.
5. Stop AI bridges.
6. Stop RViz.
7. Stop ROS nodes and roscore.
8. Restore terminal state.

Map saving can be disabled:

```bash
export AI_SAVE_MAP_ON_EXIT=0
```

## 16. Safety Rules

The system protects movement by:

- limiting command bridge max linear and angular velocity,
- stopping forward motion when lidar is missing,
- stopping forward motion when lidar obstacle distance is below the threshold,
- cancelling move_base before manual stop/search cancellation,
- sending repeated zero velocity packets during emergency stop,
- stopping search immediately on UI stop.

## 18. Troubleshooting

### Robot stops too often during search

Check that these are disabled:

```bash
export AI_SEARCH_STOP_AT_WAYPOINT=0
export AI_SEARCH_SCAN_SECONDS=0
export AI_SEARCH_START_SCAN_SECONDS=0
export AI_SEARCH_FULL_SCAN_EVERY=0
export AI_SEARCH_ROTATE_ON_STUCK=0
export AI_SEARCH_MAX_RECOVERIES_PER_WP=0
```

### Object is visible but not found

Check:

- camera bridge log: `/tmp/ros_rgb_camera_ai_bridge.log`
- AI log: `/tmp/ai_companion.log`
- YOLO confidence:

```bash
export AI_YOLO_CONF=0.20
```

Lower values detect more objects but may increase false positives.

### RViz button opens a new RViz instead of showing old one

Install one of the window-control tools on the host:

```bash
sudo apt-get install -y wmctrl xdotool
```

The button still detects existing RViz processes, but raising a graphical window depends on the host X11 window manager tools.

### Lidar keeps spinning after CTRL+C

The cleanup path calls lidar stop services and kills lidar drivers. If hardware still spins, check Docker services:

```bash
docker exec -it yahboom_container /bin/bash -lc "rosservice list | grep -E 'stop_motor|stop_lidar|stop_scan'"
```

### AMCL pose is not available

Check:

```bash
docker exec -it yahboom_container /bin/bash -lc "rostopic echo /amcl_pose -n 1"
```

Also verify that startup completed the AMCL global localization and spin.

## 20. Design Principle

The stable search behavior should be:

- navigate through map waypoints,
- watch camera live while moving,
- stop only when target is detected or user cancels,
- skip bad waypoints without manual spinning,
- avoid wasting time at every waypoint,
- keep RViz visible for operator inspection,
- shut down hardware safely.
