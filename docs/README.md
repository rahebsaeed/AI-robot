## 1. System Goal

The project combines:

- Yahboom ROS navigation running inside `yahboom_container`.
- AMCL localization on a saved map, usually `salle_robotique`.
- Lidar obstacle data from `/scan`.
- RGB camera frames from `/camera/rgb/image_raw`.
- A Python AI companion on the host in `/home/jetson/AI/main.py`.
- YOLO object detection for visual search.
- A Qt robot face interface with buttons for stop search, RViz, and microphone mute.
- UDP bridges between ROS-in-Docker and the host AI process.

The core user workflow is:

1. Start `start_navigation.sh`.
2. The script starts ROS, localization, navigation, bridges, RViz, and the AI.
3. The user gives voice commands.
4. The AI either answers conversationally, moves the robot, navigates to saved places, or searches the map for objects.
5. Search uses map waypoints and live camera detection.
6. CTRL+C stops AI, robot motion, lidar drivers, ROS nodes, and saves the map.

## 3. Runtime Architecture

The architecture has two main execution zones:

### Host / Jetson AI Zone

The host runs:

- `main.py`
- `robot_face.py`
- Whisper speech recognition
- YOLO inference
- UDP listeners for camera, lidar, and AMCL pose
- UDP sender for movement commands

### Docker / ROS Zone

The Docker container runs:

- `roscore`
- Yahboom hardware bringup
- Lidar driver
- Astra camera ROS stream
- `map_server`
- `amcl`
- `move_base`
- UDP bridge scripts generated by `start_navigation.sh`
- RViz when launched inside the container

The two zones communicate through UDP and ROS topics.

| `5020` | Host to Docker | AI movement command bridge to `/cmd_vel`. |

## 5. Launch Sequence

`start_navigation.sh` performs this sequence:

1. Defines map, Docker container, robot IP, map path, and AI path.
2. Stops old AI and bridge processes.
3. Prepares X11 permissions for RViz and the face UI.
4. Restarts robot hardware service.
5. Restarts `yahboom_container`.
6. Syncs device links inside Docker:
   - `/dev/camera_head`
   - `/dev/camera_depth`
   - `/dev/camera_usb`
   - `/dev/ydlidar`
   - `/dev/myserial`
   - `/dev/myspeech`
7. Stops old ROS processes.
8. Checks that the map YAML exists.
9. Starts `roscore`.
10. Starts Yahboom body, lidar, and Astra camera launch.
11. Starts navigation with the selected map.
12. Verifies `/scan`, `/map`, `/tf`, AMCL, and move_base services.
13. Validates usable laser ranges and the `odom -> base -> laser` TF chain.
14. Seeds AMCL from a configured or saved pose when available; otherwise starts
    global localization.
15. Runs up to three alternating full rotations and requests AMCL no-motion
    updates between attempts.
16. Requires covariance convergence and a valid `map -> base` transform.
17. Prints the verified AMCL pose and clears move_base costmaps.
18. Waits for `/camera/rgb/image_raw`.
19. Starts UDP bridges:
    - lidar bridge
    - command bridge
    - RGB camera bridge
    - pose bridge
20. Starts RViz.
21. Starts the AI companion.

### AMCL startup configuration

For a robot that always starts at a known dock, configure the dock pose in map
coordinates. Yaw is in radians:

```bash
export AI_AMCL_INITIAL_X=1.20
export AI_AMCL_INITIAL_Y=-0.45
export AI_AMCL_INITIAL_YAW=1.57
./start_navigation.sh salle_robotique
```

As soon as AMCL converges, and again on a clean CTRL+C shutdown, the launcher
saves the pose in `config/last_amcl_pose.json` and uses it on the next startup. Set
`AI_AMCL_USE_SAVED_POSE=0` if the robot was physically moved while powered off.

| `AI_AMCL_REFINEMENT_SPIN_SECONDS` | `5` | Short follow-up scan that retains the existing particle cloud. |
| `AI_AMCL_SPIN_ANGULAR` | `0.65` | Rotation speed in radians per second. |

## 6. ROS Bridge Details

### Lidar Bridge

The lidar bridge subscribes to `/scan`, extracts valid ranges, and sends:

```json
{
  "front": 0.82,
  "all": 0.61
}
```

The host AI receives this on UDP port `5010`.

### Command Bridge

The command bridge receives host UDP commands on port `5020`:

```json
{
  "type": "move",
  "direction": "forward",
  "speed": 0.25
}
```

It converts that to `/cmd_vel` messages with capped linear and angular velocity.

### RGB Camera Bridge

The RGB bridge subscribes to `/camera/rgb/image_raw`, resizes frames to `320x240`, JPEG-encodes them, and sends packets to UDP port `5030`.

### Pose Bridge

The pose bridge subscribes to `/amcl_pose`, converts quaternion orientation to yaw, and sends:

```json
{
  "x": -1.25,
  "y": 4.80,
  "yaw": 1.57
}
```

The host AI caches this pose for fast distance checks.

## 7. Perception System

`core/perceptions.py` handles:

- Speech-to-text with Whisper.
- Microphone selection and calibration.
- UDP camera frame reception.
- UDP lidar distance reception.
- YOLO object detection.
- Text-to-speech with `espeak-ng`.

### Speech Recognition and Noise Handling

Whisper is selected automatically after Qwen has loaded so the STT model fits
the remaining CUDA memory. Automatic mode only considers complete models that
are already cached, so a missing large checkpoint cannot block robot startup:

1. `turbo` when at least about 7 GiB is free.
2. `small.en` when at least about 2.7 GiB is free.
3. `base.en` when at least about 1.3 GiB is free.
4. `tiny.en` as the low-memory fallback.

The speech pipeline keeps SpeechRecognition's dynamic energy threshold active.
It performs an initial ambient-noise calibration, refreshes that calibration
periodically, records mono audio, resamples to 16 kHz, and rejects clips that
are too short or too quiet. Whisper uses deterministic beam-search decoding and
the result is rejected when its average log probability is too low or its
`no_speech` probability indicates noise. A short cooldown after `espeak-ng`
prevents the robot from transcribing its own voice.

Important environment variables:

| Variable | Default | Purpose |
| --- | --- | --- |
| `AI_WHISPER_MODEL` | `auto` | Force `turbo`, `small.en`, `base.en`, or another installed official model. |
| `AI_WHISPER_AUTO_DOWNLOAD` | `0` | Allow automatic mode to download a missing model during startup. Keep disabled for reliable launch. |
| `AI_WHISPER_LANGUAGE` | `en` | Skip language detection and transcribe as English. |
| `AI_WHISPER_BEAM_SIZE` | `5` | Beam width; lower values reduce latency. |
| `AI_MIC_CALIBRATION_SECONDS` | `1.5` | Initial quiet-room calibration duration. |
| `AI_MIC_RECALIBRATION_SECONDS` | `45` | Interval for short ambient-noise recalibration. |
| `AI_MIC_DYNAMIC_RATIO` | `1.8` | Required energy above the estimated noise floor. |
| `AI_MIC_ENERGY_THRESHOLD` | unset | Optional fixed threshold; disables automatic recalibration. |
| `AI_STT_MIN_RMS` | `35` | Reject very weak captured audio before inference. |
| `AI_STT_REJECT_NO_SPEECH` | `0.72` | Reject probable non-speech segments. |
| `AI_STT_REJECT_LOGPROB` | `-1.15` | Reject very low-confidence transcriptions. |
| `AI_STT_ECHO_COOLDOWN` | `0.45` | Delay listening after robot speech. |

For English robot commands, `small.en` is the stable manual choice when `turbo`
does not fit alongside Qwen and YOLO:

```bash
AI_WHISPER_MODEL=small.en ./start_navigation.sh salle_robotique
```

Download a larger model separately before using automatic mode. This avoids
leaving the full AI application blocked while the checkpoint is transferred:

```bash
python3 -c "import whisper; whisper.load_model('small.en', device='cpu')"
```

Until that download completes, use the cached low-latency model explicitly:

```bash
AI_WHISPER_MODEL=tiny.en ./start_navigation.sh salle_robotique
```

## 8. Main AI Loop

`main.py` creates:

- `Brain`
- `Perceptions`
- `Robot`
- `FaceBridge`
- `PlacesMemory`
- `MapSearchTask`
- `UiCommandListener`

The loop does:

1. Read latest camera and lidar state.
2. Listen for a voice command.
3. Route command by priority:
   - save / rename / delete / list saved places
   - map search request
   - normal conversation or movement command
4. For search commands, bypass the language model and call `MapSearchTask.search()`.
5. For movement commands, apply safe movement rules before sending robot commands.
6. Update face UI and speech output.

## 9. Search Command Parsing

Search commands are detected by patterns such as:

- `search for`
- `look for`
- `find`
- `where is`
- `locate`
- `go find`

Examples:

| Voice command | Parsed target | Detector target |
| --- | --- | --- |
| `search for the bottle` | `bottle` | `bottle` |
| `find the bottle of water` | `bottle of water` | `bottle` |
| `where is the chair` | `chair` | `chair` |
| `find the phone` | `phone` | `cell phone` |

YOLO aliases are handled in `core/search_tasks.py`.

## 10. Map Search Algorithm

The map search algorithm uses:

- AMCL pose
- generated free-space waypoints
- move_base goals
- live YOLO camera checks
- stuck waypoint skipping
- optional local scans and waypoint checks controlled by environment variables

The current algorithm is designed to avoid wasting time at each waypoint. It checks the camera while the robot is moving. It only stops when:

- the target is detected and confirmed,
- the user cancels search,
- the whole search completes,
- shutdown is triggered,
- optional waypoint checking/scanning is enabled.

### Search Steps

1. Reset search state.
2. Validate the target text.
3. Resolve target to a YOLO class or alias.
4. Cancel any previous navigation goal.
5. Stop robot before starting new search.
6. Check current camera before waypoint generation.
7. If target is already visible, stop and report found.
8. Clear costmaps.
9. Generate search waypoints from the map.
10. Choose nearest unvisited waypoint from current AMCL pose.
11. Send a move_base goal.
12. While moving:
    - read camera detections,
    - if target appears, cancel navigation, stop, confirm detection, and report found,
    - read AMCL distance to goal,
    - if close enough, mark waypoint visited and continue,
    - if progress is stuck, mark waypoint skipped and continue.
13. Continue until target found or waypoints exhausted.
14. Report not found if all reachable points were searched.

### Stuck Waypoints

A waypoint is considered stuck when distance to the goal does not improve enough for `AI_SEARCH_STUCK_TIMEOUT`.

Default behavior:

- Do not rotate.
- Do not run manual recovery.
- Mark waypoint as skipped.
- Continue to the next waypoint.

Manual rotation can be re-enabled with:

```bash
export AI_SEARCH_ROTATE_ON_STUCK=1
export AI_SEARCH_MAX_RECOVERIES_PER_WP=1
```

## 12. Waypoint Generation

`Robot.get_search_waypoints()` reads the map YAML and PGM image from:

```text
/root/yahboomcar_ws/src/yahboomcar_nav/maps
```

It:

1. Loads map resolution and origin.
2. Loads the occupancy image.
3. Dilates occupied pixels by a clearance radius.
4. Keeps only clearly free cells.
5. Converts free pixels to map-frame coordinates.
6. Downsamples to `AI_SEARCH_MAX_POINTS`.

This keeps goals away from walls and obstacles.

## 13. Robot Navigation

`Robot.goto_map(x, y, yaw)` publishes a goal to:

```text
/move_base_simple/goal
```

The goal frame is:

```text
map
```

The robot pose is read from cached UDP AMCL pose data when available. If the UDP cache is empty, `Robot.get_amcl_pose()` falls back to a one-shot Docker ROS read.

## 14. RViz Behavior

RViz is opened automatically by `start_navigation.sh` unless disabled:

```bash
export AI_AUTO_RVIZ=0
```

The face UI has an `RViz` button. When clicked:

1. `robot_face.py` sends `show_rviz` to UDP port `5006`.
2. `main.py` receives it in `UiCommandListener`.
3. `Robot.launch_rviz()` checks if RViz is already running.
4. If RViz exists, it tries to raise the existing window.
5. If no RViz exists, it starts one.

RViz displays:

- map
- global costmap
- robot model
- AMCL pose covariance
- lidar scan
- global plan
- TF tree

## 15. Shutdown Behavior

CTRL+C in `start_navigation.sh` runs `cleanup()`:

1. Stop host AI and face interface.
2. Save a converged AMCL pose for the next startup.
3. Stop robot motion and stale `/cmd_vel` publishers.
4. Save a live SLAM map if `/map` is being published by a mapper.
5. Stop lidar motor and lidar ROS drivers.
6. Stop AI bridges and RViz.
7. Stop ROS nodes and roscore.
8. Restore terminal state.

Map saving can be disabled:

```bash
export AI_SAVE_MAP_ON_EXIT=0
```

Normal AMCL navigation reads a static map from `map_server`; it does not update
the occupancy grid. On shutdown, `start_navigation.sh` now skips map saving when
`/map` is static and prints the reason. Use `auto_scan.sh MAP_NAME` to update a
saved occupancy map. Set `AI_SAVE_STATIC_MAP_ON_EXIT=1` only when you explicitly
want to re-save the unchanged static map.

## 16. Safety Rules

The system protects movement by:

- limiting command bridge max linear and angular velocity,
- stopping forward motion when lidar is missing,
- stopping forward motion when lidar obstacle distance is below the threshold,
- cancelling move_base before manual stop/search cancellation,
- sending repeated zero velocity packets during emergency stop,
- stopping search immediately on UI stop.

## 18. Troubleshooting

### Robot stops too often during search

Check that these are disabled:

```bash
export AI_SEARCH_STOP_AT_WAYPOINT=0
export AI_SEARCH_SCAN_SECONDS=0
export AI_SEARCH_START_SCAN_SECONDS=0
export AI_SEARCH_FULL_SCAN_EVERY=0
export AI_SEARCH_ROTATE_ON_STUCK=0
export AI_SEARCH_MAX_RECOVERIES_PER_WP=0
```

Search navigation waits longer before declaring a waypoint stuck:

```bash
export AI_SEARCH_START_GRACE_SECONDS=5.0
export AI_SEARCH_STUCK_TIMEOUT=8.0
export AI_SEARCH_MAX_STUCK_GOALS=4
```

The physical Jetson microphone is muted by default so background speech and
motor noise do not create false commands during navigation. Enable it from the
Android Control tab only when needed, or override startup behavior:

```bash
export AI_ROBOT_MIC_DEFAULT_MUTED=0
export AI_MIC_LISTEN_WHILE_MOVING=1
```

### Object is visible but not found

Check:

- camera bridge log: `/tmp/ros_rgb_camera_ai_bridge.log`
- AI log: `/tmp/ai_companion.log`
- YOLO confidence:

```bash
export AI_YOLO_CONF=0.20
```

Lower values detect more objects but may increase false positives.

### RViz button opens a new RViz instead of showing old one

Install one of the window-control tools on the host:

```bash
sudo apt-get install -y wmctrl xdotool
```

The button still detects existing RViz processes, but raising a graphical window depends on the host X11 window manager tools.

### Lidar keeps spinning after CTRL+C

The cleanup path calls lidar stop services and kills lidar drivers. If hardware still spins, check Docker services:

```bash
docker exec -it yahboom_container /bin/bash -lc "rosservice list | grep -E 'stop_motor|stop_lidar|stop_scan'"
```

### AMCL pose is not available

The launcher now stops before autonomous navigation if AMCL is not trustworthy.
Read the specific `AMCL_PREFLIGHT_ERROR` or `AMCL_NOT_CONVERGED` message. Check:

```bash
docker exec -it yahboom_container /bin/bash -lc "rostopic echo /amcl_pose -n 1"
docker exec -it yahboom_container /bin/bash -lc "rostopic echo /scan -n 1"
docker exec -it yahboom_container /bin/bash -lc "rosrun tf tf_echo odom base_footprint"
```

If the robot starts at a fixed location, set `AI_AMCL_INITIAL_X`,
`AI_AMCL_INITIAL_Y`, and `AI_AMCL_INITIAL_YAW` once. A clean shutdown then
persists the converged pose for later starts.

## 19. Professional Automatic Mapping

`auto_scan.sh` uses the same Yahboom bringup and gmapping launch files as the
working keyboard scanner, then starts `tools/professional_explorer.py` instead
of `explore_lite`.

```bash
./auto_scan.sh salle_robotique
```

The coordinator:

1. Detects connected free/unknown frontier boundaries.
2. Splits long boundaries into multiple spaced exploration regions.
3. Uses `move_base` when its action server is available. If the Yahboom mapping
   stack exposes no action server, it builds collision-inflated grid routes with
   an internal A* planner and follows them using TF pose feedback and live LiDAR
   front/left/right safety sectors.
4. Stores visited goals, failed goals, and sampled robot positions in
   `/root/yahboomcar_ws/src/yahboomcar_nav/maps/.MAP_exploration.json`.
5. Rejects fake near goals that are already inside tolerance, then uses a short
   LiDAR-guarded forward probe to expand the map before selecting another goal.
6. Confirms that no reachable unvisited frontier remains, saves the map, stops
   ROS safely, and exits automatically.

CTRL+C saves a valid partial map. Running `auto_scan.sh MAP_NAME` again backs up
and removes the existing `.yaml`, `.pgm`, and exploration memory first, then
creates a fresh map with the same name. The previous map is kept as a hidden
timestamped backup in the maps folder. To keep the old files in place, run with
`AUTO_SCAN_KEEP_EXISTING=1`. To skip maps that are already marked complete:

```bash
AUTO_SCAN_SKIP_COMPLETED=1 ./auto_scan.sh salle_robotique
```

Gmapping cannot load a saved occupancy image as a resumable SLAM state. For
that reason, incomplete memory is reset when the script starts a new gmapping
session. `AUTO_SCAN_REUSE_MEMORY=1` is only appropriate when restarting the
coordinator inside the same unchanged map/odometry frame.

| Variable | Default | Meaning |
| --- | --- | --- |
| `AUTO_SCAN_MIN_FRONTIER_LENGTH` | `0.35` | Minimum frontier boundary length in metres. |
| `AUTO_SCAN_MIN_CLEARANCE` | `0.35` | Minimum goal clearance from mapped obstacles. |
| `AUTO_SCAN_VISITED_RADIUS` | `0.85` | Radius used to suppress already-scanned goals. |
| `AUTO_SCAN_PLAN_CANDIDATES` | `6` | Highest-utility candidates checked by the available service or internal A*. |

| `AUTO_SCAN_MIN_GOAL_DISTANCE` | `0.10` | Minimum normal frontier travel distance; small values allow fresh maps to start moving. |
| `AUTO_SCAN_MIN_NEAR_GOAL_DISTANCE` | `0.05` | Minimum close-frontier distance accepted during startup expansion. |
| `AUTO_SCAN_MAX_LINEAR_SPEED` | `0.35` | Default direct A* and move_base linear exploration speed. |
| `AUTO_SCAN_MAX_ANGULAR_SPEED` | `1.00` | Default direct A* and move_base angular exploration speed. |

## 21. Design Principle

The stable search behavior should be:

- navigate through map waypoints,
- watch camera live while moving,
- stop only when target is detected or user cancels,
- skip bad waypoints without manual spinning,
- avoid wasting time at every waypoint,
- keep RViz visible for operator inspection,
- shut down hardware safely.
